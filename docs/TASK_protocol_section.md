# Задача: реализовать раздел «Протокол» для конфигуратора Geotek GT-9

## Цель
Реализовать в новой системе вкладку «Протокол», которая читает и записывает в устройство битовые маски телеметрии — какие поля включаются в пакет, отправляемый на сервер. Вкладка должна быть функционально эквивалентна разделу «Протокол» в GT9_config (WinForms, Form1.cs).

## Транспорт
- Связь с устройством — последовательный порт (COM).
- Формат команды (отправка): `$<password>;<COMMAND>\r\n`, по умолчанию `password = 1234`.
- Формат ответа (приём): строка начинается с `$`, поля разделены `;`, заканчивается `\r\n`. Первое поле — идентификатор команды (например `$PRSET20`).
- Между последовательными командами выдерживать паузу ~100 мс.

## Модель данных

В устройстве две 16-байтовых битовых маски, каждая передаётся как 32-символьная HEX-строка (8 слов по 16 бит, MSB слева — позиция 0 = старший бит слова):

- **PRSET20** — основная маска (статус, GPS, питание, GSM, входы/выходы, LLS).
- **PRSET21** — дополнительная маска (ДУТ НПП датчики).

Для старого протокола (`protocolType == "0"`) вместо PRSET20/21 используются **PRSET10/PRSET11** — содержимое и биты идентичны, отличается только имя команды. `protocolType` определяется при инициализации устройства (из `$VER`/другой команды идентификации) и должен влиять на все GET/SET раздела.

### Карта битов PRSET20

| Слово | Индекс бита (0=MSB) | Поле UI | Смысл |
|---|---|---|---|
| word1 | 0 | samplenumber | номер отсчёта |
| word1 | 1 | devicestatus | статус устройства |
| word2 | 8 | hdop | HDOP GPS |
| word2 | 3 | — | константа 1 (всегда включено, см. `0x1c` комментарий) |
| word3 | 11 | internaltemp | внутренняя температура |
| word3 | 14 | acc | ACC (зажигание) |
| word3 | 15 | vext | внешнее питание |
| word4 | 13 | gsmlbs | GSM LBS |
| word4 | 14 | gsm_rssi | GSM RSSI |
| word5 | — | — | константа `0x000C` (всегда) |
| word6 | 15..8 | in1p..in8p | цифровые входы 1..8 |
| word7 | 15..12 | out1p..out4p | цифровые выходы 1..4 |
| word8 | 14..9 | lls1p..lls6p | LLS датчики 1..6 |

Все прочие биты = 0.

### Карта битов PRSET21

| Слово | Индекс бита (0=MSB) | Поле UI | Смысл |
|---|---|---|---|
| word5 | 14..9 | sensor1..sensor6 | **ДУТ НПП** датчики 1..6 |

Слова 1–4, 6–8 = `0000`.

**Важно:** `lls1p..lls6p` (PRSET20) и `sensor1..sensor6` (PRSET21) — два независимых набора каналов. Включение в одном не включает в другом.

## Команды

### Чтение (при открытии вкладки)
```
$1234;PRSET20;GET
(пауза 100 мс)
$1234;PRSET21;GET
```
Для старого протокола — `PRSET10;GET` / `PRSET11;GET`.

Ожидаемые ответы:
```
$PRSET20;C00000000003000000000F0F03030000
$PRSET21;00000000000000000002000000000000
```
Второе поле — 32 HEX-символа. Распарсить по 4 символа на слово → 16 бит MSB-first → развести по чекбоксам согласно картам выше.

### Запись (по кнопке «Применить»)
Собрать 32-символьную HEX-строку из состояния чекбоксов (обратная операция), отправить:
```
$1234;PRSET20;SET;<32 hex>
(пауза 100 мс)
$1234;PRSET21;SET;<32 hex>
```
(или `PRSET10;SET` / `PRSET11;SET` для старого протокола).

## Алгоритм кодирования слова
```
bits = строка '0'/'1' длиной 16, индекс 0 = старший бит
word_hex = IntToHex(BinToInt(bits)).PadLeft(4, '0').ToUpper()
```
Алгоритм декодирования — обратный: `bits = IntToBin(HexToInt(word_hex)).PadLeft(16, '0')`.

## UI (группы чекбоксов на вкладке)
1. **Общие:** samplenumber, devicestatus, hdop, internaltemp, acc, vext, gsmlbs, gsm_rssi.
2. **Входы:** in1..in8 (8 чекбоксов).
3. **Выходы:** out1..out4 (4 чекбокса).
4. **LLS:** lls1..lls6 (6 чекбоксов).
5. **ДУТ НПП:** sensor1..sensor6 (6 чекбоксов) — отдельная группа.
6. Кнопки **«Прочитать»** и **«Записать»**, индикатор прогресса (2 шага).

## Тест-кейсы
1. Отправить `PRSET20;GET` с ответом `C00000000003000000000F0F03030000` — проверить, что отмечаются: samplenumber, devicestatus, in1..in4, out1..out2, lls1..lls2 (сверить с картой выше на реальном устройстве).
2. Отметить только `acc` + `vext`, записать, прочитать обратно — маска должна совпадать.
3. Отметить все 6 ДУТ НПП в PRSET21, записать/прочитать — word5 = `0x7E00` (биты 14..9 = 111111).
4. Переключение `protocolType` 0↔1 должно менять префикс команды PRSET1x ↔ PRSET2x без изменения карт битов.
5. Независимость LLS и ДУТ НПП: включение lls3 не должно влиять на sensor3 и наоборот.

## Критерии приёмки
- Вкладка корректно читает маски с реального устройства GT-9 и отображает все перечисленные поля.
- Запись маски возвращается без изменений при повторном чтении (round-trip).
- Поддержаны оба варианта имён команд (PRSET10/11 и PRSET20/21) в зависимости от `protocolType`.
- Константные биты (word2 bit 3 PRSET20, word5 PRSET20 = `0x000C`) всегда выставляются при записи, независимо от UI.
